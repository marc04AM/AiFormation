SKILL-AUDIT.md — l'audit sorgente completo con esito per ogni file

Contesto

Il marketplace Sistec è interno. La specificità di dominio (schema language_spv, Sistec.Core, convenzioni HMI) non squalifica un file — anzi, è il valore specifico che una skill porta al team. I command sono deprecati a favore delle skill: la forma giusta è skill esplicita con disable-model-invocation: true dove l'operazione va invocata a mano.

§1

Il filtro — le tre domande che decidono

Un file merita una skill nel marketplace solo se passa tutte e tre:

  1. Lo fa già, meglio, uno strumento deterministico? (git, linter/analyzer, docker compose, uno script). Se sì, un'approssimazione LLM ha valore negativo: aggiunge non-determinismo e rischio, zero guadagno.
  2. È solo plan-mode / TodoWrite / comportamento nativo travestito? Se sì, è ridondante — Claude lo fa già senza skill.
  3. Claude farebbe materialmente peggio SENZA? Deve esserci giudizio semantico reale che solo un LLM porta: interpretare un output, tradurre intento, refactor semantico.

Il pattern che vince

Uno script deterministico raccoglie dati, poi l'LLM interpreta/decide (es. dpi-anchor-fix, network-probe). Oppure l'LLM fa un lavoro puramente semantico che nessun tool sa fare (es. maintain-manual, translate, reconcile-solutions).

Qualità del porting

Ogni skill è stata de-coupled dal workspace privato: path assoluti C:\Users\Sistec 23\… → $env:CLAUDE_PLUGIN_ROOT o relativi; direttive proprietarie rimosse; descrizione riscritta orientata al trigger; engine deterministici (.ps1) bundlati come assets/ del plugin. Barra di qualità: add-doc (corta, generale, zero leakage).

§3

Skill rifiutate (⚫) — perché

Quattordici file sono stati scartati. Le motivazioni si raggruppano in quattro categorie:

3.1 Operazioni deterministiche — le fa meglio uno script

Un LLM è non-deterministico per natura. Dove esiste un tool deterministico (git, linter, docker compose, script di copia), usare un LLM aggiunge rischio senza beneficio.

FileMotivo
fetchGit puro (fetch, rev-list, status --porcelain, merge --ff-only) + predicati. Un loop/script lo fa meglio, in modo deterministico e senza consumare contesto.
gitAlignGit puro + predicati. L'unico valore (guard "no --split a merge in corso") è un check da mettere in gitize, non una skill.
gitWipeGit puro (for-each-ref, branch --merged, branch -d/-D). Operazione distruttiva → va fatta da uno script deterministico, non da un LLM. Safety net = git stesso.
mergegit merge --no-ff in loop. L'unico bit LLM (comporre il messaggio) è già il mestiere di gitize. «Non basta git merge?»
codesysWrapper sottile su docker compose (up/down/restart = comandi compose 1:1). Modo nativo: docker compose.
deployRilascio in produzione (copia UNC + shortcut): operazione rischiosa, irreversibile, deterministica. 240 righe per un copia-incolla che un script (robocopy + WScript.Shell) fa meglio e in sicurezza.
deadcodeScan LLM dell'intera codebase = spreco di contesto. Il dead code lo trovano linter/analyzer (Roslyn IDE0051/CS0169, ReSharper) in modo deterministico e completo.

3.2 Comportamento nativo — Claude lo fa già

Alcuni file cercano di insegnare a Claude ciò che Claude sa fare nativamente. Sono ridondanti.

FileMotivo
projInfrastruttura di sessione. Nativo CC = lanciare claude dentro ogni repo (la cwd è il progetto). Framework privato (_config, MEMORY.md, anchor, work-dir stale) non riusabile.
feedbackVersione striminzita della plan mode nativa di Claude Code.
issueIssue-tracker su file. Un umano apre su GitHub; l'analisi è nativa («analizza questo problema»); i verbi CRUD = manipolazione deterministica di tabella markdown; persistenza = spec-driven / «salva un md».
port-solutionRimodella un tipo su un esemplare di un'altra solution. Ma il workflow giusto per una feature è un branch (contesto ampio, git merge, meno rischio API pubbliche), non copie parallele. Tolto il framing «due solution» = refactor nativo.

3.3 Meta-tooling — coperto da skill-creator

Strumenti per creare o gestire skill. Sono già coperti dallo skill-creator del marketplace.

FileMotivo
commandMeta-tooling. Nativo = creare .claude/commands/<name>.md; per distillare skill c'è skill-creator. Spawner/ledger/catalog bespoke non riusabili.
skillifyMeta-authoring. Coperto da skill-creator; «recording» solo comportamentale (CC non ha hook on-disk) → fragile.
skilliDoPartner-produttore di skillify. Stesso motivo.

Il criterio che ha salvato reconcile-solutions

reconcile-solutions e port-solution nascono dall'abitudine di tenere copie parallele di solution (es. 5309 e 5315). Il workflow migliore è un branch — ma quando i prodotti sono davvero distinti (non branch dello stesso repo), l'LLM è l'unico modo per riapplicare l'intento della feature. port-solution è stato scartato perché nel caso comune si riduce a refactor nativo con branch; reconcile-solutions è stato tenuto per il caso raro di heritage condiviso tra prodotti distinti.
